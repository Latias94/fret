# Layout Engine v2 Migration Inventory

This document is a living checklist for migrating from container-owned "Taffy islands" to the
window-scoped layout engine (`layout-engine-v2`), with multi-viewport docking as the end-state.

It is **not** an ADR. Contracts live in:

- `docs/adr/0115-available-space-and-non-reentrant-measurement.md`
- `docs/adr/0116-window-scoped-layout-engine-and-viewport-roots.md`

Related roadmap:

- `docs/layout-engine-refactor-roadmap.md`

## Status Values

- **Done**: implemented and covered by tests (in this worktree/branch unless otherwise noted).
- **In progress**: partially migrated; gaps remain.
- **Planned**: agreed direction; not implemented.
- **Intentional barrier**: should remain a specialized algorithm, but must interop cleanly.

## Migration Checklist

| Area / Barrier | v2 status | Notes | Tests / Evidence |
| --- | --- | --- | --- |
| Viewport roots (multi-viewport docking) | Done | Each viewport is solved as an independent root; percent/flex only resolve against definite per-viewport available space. Multi-window geometry queries (`bounds_for_element`) are window-scoped and must not leak across tear-off windows. | Conformance tests in `crates/fret-ui/src/declarative/tests/layout.rs` (viewport-root suite, including nested registration flush); docking integration: `ecosystem/fret-ui-docking/src/dock/tests.rs` (notably `bounds_for_element_is_window_scoped_across_multi_window_docking` and `overlay_placement_must_use_window_local_anchor_bounds`, run with `-p fret-ui-docking --features layout-engine-v2`). |
| Flow wrappers (Container/Pressable/Opacity/VisualTransform/Semantics/FocusScope/Stack/InteractivityGate/PointerRegion/HoverRegion/WheelRegion/Anchored) | In progress | Prefer building these into the same Taffy tree when layout-style-only and static; keep `measure_in` for true leaves only. Wrapper/positioned probe sizing should use `measure_in` (not `layout_in`) to avoid re-entrant subtree layout. Multi-child wrapper boxes are progressively represented in the engine tree (overlay-style). Fallback wrapper sizing/layout shares `layout_positioned_container_impl` so probe rect budgets and absolute-child probing stay consistent across wrappers and Stack. Containers that lay out children in definite rects also precompute child flow islands in Final passes (e.g. Container, Anchored, ResizableSplit). Flex/Grid root-style solves are centralized via `UiTree::solve_flow_island_with_root_style`; after solving, they only precompute child flow islands when the child subtree is not already engine-backed (barrier interop) via `UiTree::flow_subtree_is_engine_backed`, reducing the number of nested Flex/Grid self-solves and avoiding redundant precompute in positioned wrapper containers. | Conformance: `declarative::tests::layout::pressable_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::stack_does_not_stretch_spacer_children_in_engine_tree`, `declarative::tests::layout::container_does_not_stretch_spacer_child_in_engine_tree`, `declarative::tests::layout::pressable_does_not_stretch_spacer_child_in_engine_tree`, `declarative::tests::layout::opacity_does_not_stretch_spacer_child_in_engine_tree`, `declarative::tests::layout::visual_transform_does_not_stretch_spacer_child_in_engine_tree` in `crates/fret-ui/src/declarative/tests/layout.rs`. Viewport-root coverage (wrapper nodes exist in engine tree): `declarative::tests::layout::viewport_root_pressable_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::viewport_root_semantics_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::viewport_root_focus_scope_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::viewport_root_opacity_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::viewport_root_visual_transform_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::viewport_root_interactivity_gate_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::viewport_root_container_wraps_multiple_children_in_engine_tree`, and `declarative::tests::layout::viewport_root_stack_wraps_multiple_children_in_engine_tree` in `crates/fret-ui/src/declarative/tests/layout.rs`. Evidence: `crates/fret-ui/src/declarative/host_widget/layout.rs` (v2 fast path helper: `try_layout_children_from_engine_or_manual_absolute`) + `crates/fret-ui/src/declarative/host_widget/layout/positioned_container.rs` + `crates/fret-ui/src/resizable_split/widget.rs` + `crates/fret-ui/src/declarative/host_widget/layout/flex.rs` + `crates/fret-ui/src/declarative/host_widget/layout/grid.rs` + `crates/fret-ui/src/tree/layout.rs` + `crates/fret-ui/src/tree/mod.rs`. Engine builder: `crates/fret-ui/src/layout_engine/flow.rs` (wrapper overlay nodes, `passthrough_wrapper_child`). |
| Scroll (barrier) | Done | Content extent is measured via `measure_in` + `AvailableSpace::MaxContent` (no `1e9` probe). Child subtrees precompute flow islands when not a probe layout, but skip precompute when the subtree is clean / translation-only (scrolling). | `crates/fret-ui/src/declarative/host_widget/layout/scrolling.rs`; regression: `declarative::tests::layout::scroll_translation_does_not_force_layout_engine_solves` in `crates/fret-ui/src/declarative/tests/layout.rs`. |
| VirtualList (barrier + virtualization) | Done | Measured mode uses `measure_in` + `MaxContent` in v2, then precomputes a flow island per visible item root before final layout; precompute is gated by `_if_needed` so translation-only scrolling does not re-solve visible roots. | `declarative::tests::virtual_list_wraps_visible_items_in_engine_tree` in `crates/fret-ui/src/declarative/tests/virtual_list.rs`. |
| ResizablePanelGroup (barrier + custom layout) | Done | After splitting the viewport into definite panel rects, v2 registers each panel root as a viewport root so the engine solves each panel subtree independently (no probe-time precompute). | `declarative::tests::interactions::resizable_panel_group_registers_viewport_roots_for_panels` in `crates/fret-ui/src/declarative/tests/interactions.rs`; conformance: `declarative::tests::layout::resizable_panel_group_does_not_register_viewport_roots_during_probe_layout` and `declarative::tests::layout::resizable_panel_group_viewport_roots_match_panel_bounds` in `crates/fret-ui/src/declarative/tests/layout.rs`. |
| Overlay roots / dismissible layers (barrier) | Done | Overlay roots are represented as flow wrapper nodes in the engine tree so leaf-only intrinsic measurement remains non-reentrant; when the overlay root is not already engine-backed, it precomputes its own flow island and then applies child rects via v2 engine fast paths. | `crates/fret-ui/src/declarative/host_widget/layout.rs` (DismissibleLayer) and `declarative::tests::layout::overlay_root_dismissible_layer_precomputes_child_flow_islands` in `crates/fret-ui/src/declarative/tests/layout.rs`. |
| Positioned containers (PointerRegion/WheelRegion multi-child) | Done | These containers can be represented as overlay-style wrappers in the engine tree (so viewport-root solves can keep intrinsic measurement leaf-only). As a fallback (or when the parent island is not solved), v2 still precomputes flow islands for non-absolute children so subtrees can use engine fast paths. Probe sizing uses `measure_in` (no subtree `layout_in` during probe). When a wrapper has only absolute children, its single-cell grid track uses `flex(1)` so `inset: 0` absolute fills use the full wrapper bounds (not a collapsed `auto` track). | `declarative::tests::layout::positioned_container_precomputes_flow_islands_for_multiple_children`, `declarative::tests::layout::viewport_root_pointer_region_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::viewport_root_pointer_region_absolute_child_fills_region`, `declarative::tests::layout::viewport_root_wheel_region_wraps_multiple_children_in_engine_tree`, and `declarative::tests::layout::viewport_root_wheel_region_absolute_child_fills_region` in `crates/fret-ui/src/declarative/tests/layout.rs`; engine behavior in `crates/fret-ui/src/layout_engine/flow.rs`. |
| HoverRegion (multi-child) | Done | Multi-child hover regions now precompute flow islands for non-absolute children, enabling engine-backed flow inside hover tracking regions. When the hover region has no absolute children, it is also represented as an overlay-style wrapper node in the engine tree so viewport root solves can keep measurement leaf-only. Probe sizing uses `measure_in` (no subtree `layout_in` during probe). | `declarative::tests::layout::viewport_root_hover_region_wraps_flow_in_engine_tree` and `declarative::tests::layout::hover_region_precomputes_flow_islands_for_multiple_children` in `crates/fret-ui/src/declarative/tests/layout.rs`. |
| Layout pass kind (probe/final) | Done | Probe layouts are an explicit phase (`LayoutPassKind::Probe` / `Final`) rather than a size-based heuristic. Probe passes must not consume one-shot state, must not register viewport roots, and should avoid expensive precomputation. Flow containers in v2 avoid mutating the layout engine during probe by returning intrinsic size via `measure_in` (e.g. Flex/Grid). | Search for `LayoutPassKind::Probe` callsites; key behavior is enforced by `LayoutCx::layout_viewport_root` and barrier widgets (scroll/virtualization), plus Flex/Grid probe guards: `crates/fret-ui/src/declarative/host_widget/layout/flex.rs`, `crates/fret-ui/src/declarative/host_widget/layout/grid.rs`. |
| Remaining `1e9` usage (code) | Done | v2 paths must not approximate `Min/MaxContent` as huge definite bounds. No remaining `1e9`-style probe constants are used in runtime code; historical probe semantics are now represented explicitly via `LayoutPassKind` rather than size heuristics. | Run the audit commands below. |

## Audit Commands

Run these periodically (and update this file with new hits):

- Find large-probe patterns (should be test-only): `rg -n "Px\\(1\\.0e9\\)|\\b1\\.0e9\\b|\\b1e9\\b" crates/fret-ui/src docs -S`
- Find remaining v2-incompatible probe layout calls: `rg -n "layout_in\\(.*Px\\(1\\.0e9\\)" crates/fret-ui/src -S`

## Notes (What We Intentionally Do Not Support)

- No general-purpose "display: contents" or Radix `Slot/asChild` prop merging (see `docs/adr/0117-trigger-composition-and-no-slot-aschild.md`).
