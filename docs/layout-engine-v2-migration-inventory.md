# Layout Engine v2 Migration Inventory

This document is a living checklist for migrating from container-owned "Taffy islands" to the
window-scoped layout engine (`layout-engine-v2`), with multi-viewport docking as the end-state.

It is **not** an ADR. Contracts live in:

- `docs/adr/0113-available-space-and-non-reentrant-measurement.md`
- `docs/adr/0114-window-scoped-layout-engine-and-viewport-roots.md`

Related roadmap:

- `docs/layout-engine-refactor-roadmap.md`

## Status Values

- **Done**: implemented and covered by tests (in this worktree/branch unless otherwise noted).
- **In progress**: partially migrated; gaps remain.
- **Planned**: agreed direction; not implemented.
- **Intentional barrier**: should remain a specialized algorithm, but must interop cleanly.

## Migration Checklist

| Area / Barrier | v2 status | Notes | Tests / Evidence |
| --- | --- | --- | --- |
| Viewport roots (multi-viewport docking) | Done | Each viewport is solved as an independent root; percent/flex only resolve against definite per-viewport available space. Viewport roots participate in the same request/build phase as layer roots so stable node identity is preserved even when a viewport root is skipped for compute/apply (clean subtree). Multi-window geometry queries (`bounds_for_element`) are window-scoped and must not leak across tear-off windows. | Conformance tests in `crates/fret-ui/src/declarative/tests/layout.rs` (viewport-root suite, including nested registration flush); regression: `declarative::tests::layout::viewport_root_request_build_keeps_engine_nodes_alive_when_skipped`; docking integration: `ecosystem/fret-docking/src/dock/tests.rs` (notably `bounds_for_element_is_window_scoped_across_multi_window_docking` and `overlay_placement_must_use_window_local_anchor_bounds`, run with `-p fret-docking`). |
| Flow wrappers (Container/Pressable/Opacity/VisualTransform/RenderTransform/Semantics/FocusScope/Stack/InteractivityGate/PointerRegion/WheelRegion/Anchored) | Done | Prefer building these into the same Taffy tree when layout-style-only and static; keep `measure_in` for true leaves only. Wrapper/positioned probe sizing should use `measure_in` (not `layout_in`) to avoid re-entrant subtree layout. Multi-child wrapper boxes are represented in the engine tree (overlay-style). In v2, wrappers do not proactively solve child subtrees during their own layout; instead, the window-scoped request/build phase keeps identity stable, and compute/apply is driven by roots (layer roots + viewport roots). Escape hatch: explicit barriers may solve a child subtree root via `LayoutCx::solve_barrier_child_root_if_needed(...)` when they lay out children into definite rects. | Conformance: `declarative::tests::layout::pressable_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::stack_does_not_stretch_spacer_children_in_engine_tree`, `declarative::tests::layout::container_does_not_stretch_spacer_child_in_engine_tree`, `declarative::tests::layout::pressable_does_not_stretch_spacer_child_in_engine_tree`, `declarative::tests::layout::opacity_does_not_stretch_spacer_child_in_engine_tree`, `declarative::tests::layout::visual_transform_does_not_stretch_spacer_child_in_engine_tree` in `crates/fret-ui/src/declarative/tests/layout.rs`. Viewport-root coverage (wrapper nodes exist in engine tree): `declarative::tests::layout::viewport_root_pressable_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::viewport_root_semantics_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::viewport_root_focus_scope_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::viewport_root_opacity_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::viewport_root_visual_transform_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::viewport_root_interactivity_gate_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::viewport_root_container_wraps_multiple_children_in_engine_tree`, and `declarative::tests::layout::viewport_root_stack_wraps_multiple_children_in_engine_tree` in `crates/fret-ui/src/declarative/tests/layout.rs`. Evidence: v2 root orchestration in `crates/fret-ui/src/tree/layout.rs`; wrapper engine nodes in `crates/fret-ui/src/layout_engine/flow.rs`; wrapper layout fast-paths in `crates/fret-ui/src/declarative/host_widget/layout.rs`. |
| Scroll (barrier) | Done | Content extent is measured via `measure_in` + `AvailableSpace::MaxContent` (no `1e9` probe). The request/build phase still traverses the mounted subtree (detached) so engine identity remains stable across frames; compute is skipped when the subtree is clean / translation-only (scrolling). | `crates/fret-ui/src/declarative/host_widget/layout/scrolling.rs`; engine request/build: `crates/fret-ui/src/layout_engine/flow.rs`; regression: `declarative::tests::layout::scroll_translation_does_not_force_layout_engine_solves` in `crates/fret-ui/src/declarative/tests/layout.rs`. |
| VirtualList (barrier + virtualization) | Done | Measured mode uses `measure_in` + `MaxContent` in v2, then precomputes a flow island per visible item root before final layout; precompute is gated by `_if_needed` so translation-only scrolling does not re-solve visible roots. The request/build phase still traverses the mounted visible subtree (detached) to keep stable engine identity across frames. | `declarative::tests::virtual_list_wraps_visible_items_in_engine_tree` in `crates/fret-ui/src/declarative/tests/virtual_list.rs`; engine request/build: `crates/fret-ui/src/layout_engine/flow.rs`. |
| ResizablePanelGroup (barrier + custom layout) | Done | After splitting the viewport into definite panel rects, v2 registers each panel root as a viewport root so the engine solves each panel subtree independently (no probe-time precompute). The request/build phase still traverses mounted panel subtrees (detached) so engine identity remains stable across frames even when panels are clean. | `declarative::tests::interactions::resizable_panel_group_registers_viewport_roots_for_panels` in `crates/fret-ui/src/declarative/tests/interactions.rs`; conformance: `declarative::tests::layout::resizable_panel_group_does_not_register_viewport_roots_during_probe_layout` and `declarative::tests::layout::resizable_panel_group_viewport_roots_match_panel_bounds` in `crates/fret-ui/src/declarative/tests/layout.rs`; engine request/build: `crates/fret-ui/src/layout_engine/flow.rs`. |
| Overlay roots / dismissible layers (barrier) | Done | Overlay roots are represented as flow wrapper nodes in the engine tree so leaf-only intrinsic measurement remains non-reentrant. In v2, overlay wrapper nodes rely on root-driven request/build + compute/apply, and do not proactively precompute their own flow islands during widget layout. | Wrapper engine nodes: `crates/fret-ui/src/layout_engine/flow.rs` (includes `DismissibleLayer` in overlay wrappers); layout fast paths: `crates/fret-ui/src/declarative/host_widget/layout.rs`; conformance: `declarative::tests::layout::overlay_root_dismissible_layer_precomputes_child_flow_islands` in `crates/fret-ui/src/declarative/tests/layout.rs`. |
| Positioned containers (PointerRegion/WheelRegion multi-child) | Done | These containers are represented as overlay-style wrapper nodes in the engine tree (so viewport-root solves can keep intrinsic measurement leaf-only). Probe sizing uses `measure_in` (no subtree `layout_in` during probe). When a wrapper has only absolute children, its single-cell grid track uses `flex(1)` so `inset: 0` absolute fills use the full wrapper bounds (not a collapsed `auto` track). | `declarative::tests::layout::positioned_container_precomputes_flow_islands_for_multiple_children`, `declarative::tests::layout::viewport_root_pointer_region_wraps_multiple_children_in_engine_tree`, `declarative::tests::layout::viewport_root_pointer_region_absolute_child_fills_region`, `declarative::tests::layout::viewport_root_wheel_region_wraps_multiple_children_in_engine_tree`, and `declarative::tests::layout::viewport_root_wheel_region_absolute_child_fills_region` in `crates/fret-ui/src/declarative/tests/layout.rs`; engine behavior in `crates/fret-ui/src/layout_engine/flow.rs`. |
| HoverRegion (multi-child) | Done | When the hover region has no absolute children, it is represented as an overlay-style wrapper node in the engine tree so viewport root solves can keep measurement leaf-only. Probe sizing uses `measure_in` (no subtree `layout_in` during probe). HoverRegion with absolute children remains a special case (it is not represented as an overlay wrapper in the engine tree) so it can size to include absolute children without coupling into the parent's flow solve. | `declarative::tests::layout::viewport_root_hover_region_wraps_flow_in_engine_tree` and `declarative::tests::layout::hover_region_precomputes_flow_islands_for_multiple_children` in `crates/fret-ui/src/declarative/tests/layout.rs`; engine behavior in `crates/fret-ui/src/layout_engine/flow.rs` (`hover_region_has_absolute_child`). |
| Layout pass kind (probe/final) | Done | Probe layouts are an explicit phase (`LayoutPassKind::Probe` / `Final`) rather than a size-based heuristic. Probe passes must not consume one-shot state, must not register viewport roots, and should avoid expensive precomputation. Probe passes also must not finalize/prune the layout engine graph (`end_frame` is Final-only) so stable node identity is preserved. Flow containers in v2 avoid mutating the layout engine during probe by returning intrinsic size via `measure_in` (e.g. Flex/Grid). | Search for `LayoutPassKind::Probe` callsites; key behavior is enforced by `LayoutCx::layout_viewport_root` and barrier widgets (scroll/virtualization), plus Flex/Grid probe guards: `crates/fret-ui/src/declarative/host_widget/layout/flex.rs`, `crates/fret-ui/src/declarative/host_widget/layout/grid.rs`; regression: `declarative::tests::layout::probe_layout_does_not_prune_layout_engine_nodes` in `crates/fret-ui/src/declarative/tests/layout.rs`. |
| Remaining `1e9` usage (code) | Done | v2 paths must not approximate `Min/MaxContent` as huge definite bounds. No remaining `1e9`-style probe constants are used in runtime code; historical probe semantics are now represented explicitly via `LayoutPassKind` rather than size heuristics. | Run the audit commands below. |

## Audit Commands

Run these periodically (and update this file with new hits):

- Find large-probe patterns (should be test-only): `rg -n "Px\\(1\\.0e9\\)|\\b1\\.0e9\\b|\\b1e9\\b" crates/fret-ui/src docs -S`
- Find remaining v2-incompatible probe layout calls: `rg -n "layout_in\\(.*Px\\(1\\.0e9\\)" crates/fret-ui/src -S`

## Notes (What We Intentionally Do Not Support)

- No general-purpose "display: contents" or Radix `Slot/asChild` prop merging (see `docs/adr/0115-trigger-composition-and-no-slot-aschild.md`).
