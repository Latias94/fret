## Why this note exists

The older parity audit answered a larger historical question:

> Was the in-tree `imui` stack directionally correct at all?

The newer closeout docs answered several follow-up questions:

> Did we close editor composites, tooltip/tree helpers, typed drag/drop seams, sortable recipes,
> and ghost choreography?

This note answered the next narrower question:

> Relative to Dear ImGui and egui, what high-frequency immediate authoring vocabulary is still
> missing after those closures?

## Current conclusion

The direction is correct.
The stack is no longer missing major runtime plumbing.

The workstream-level conclusion is now closed:

- canonical `selectable`, `combo`, `table`, `virtual_list`, and `separator_text` immediate surfaces
  all landed in `fret-ui-kit::imui`,
- the owner split remained clean (`imui` generic helpers vs editor composites vs shell policy),
- and no active generic immediate vocabulary gaps remain in this workstream.

## Current shipped floor

Today `imui` is already strong enough in several areas:

- layout containers: `horizontal`, `vertical`, `grid`, `scroll`
- popup/menu surfaces
- tooltip surfaces
- floating areas/windows
- buttons, toggles, sliders, text input helpers
- tree/collapsing helpers
- typed drag/drop helpers
- editor immediate adapters and inspector composites

Evidence:

- `ecosystem/fret-ui-kit/src/imui.rs`
- `ecosystem/fret-ui-editor/src/imui.rs`
- `apps/fret-examples/src/imui_editor_proof_demo.rs`

Practical implication:

- the next lane should not spend its main budget on more button/text-input-level wrappers,
- and should not reopen already-closed generic helper questions.

## P0 gaps

### A. Generic `selectable` row/item family

Why it is still a gap:

- Dear ImGui uses `Selectable()` as a core row primitive for lists, menus, popups, and simple
  multi-item panes.
- egui uses `ui.selectable_value(...)` and `SelectableLabel` in the same role.
- Fret currently has menu rows, tree rows, and editor-specific selection flows, but no single
  generic immediate row-selection helper that call sites can reuse broadly.

Current consequence:

- authoring code keeps re-solving selected row chrome and hit-box policy piecemeal,
- popup option lists gravitate toward menu-item semantics even when "menu" is not the right model,
- and future combo/table/clipper helpers lack a canonical row primitive to build on.

Recommended owner:

- `ecosystem/fret-ui-kit::imui`

Upstream anchors:

- Dear ImGui: `repo-ref/imgui/imgui.h` (`Selectable`), `repo-ref/imgui/imgui_widgets.cpp`
- egui: `repo-ref/egui/crates/egui/src/widgets/selected_label.rs`, frequent
  `ui.selectable_value(...)` call sites

### B. Generic immediate `begin_combo` / `combo` family

Why it is still a gap:

- Dear ImGui treats `BeginCombo()` as a generic preview + popup body pattern.
- egui exposes `ComboBox` as a routine immediate picker.
- Fret's older `select_model` helper was a narrow popup-radio convenience, not a general combo
  surface.

Current evidence:

- `ecosystem/fret-ui-kit/src/imui/select_controls.rs`
- `ecosystem/fret-ui-kit/src/imui/combo_controls.rs`

What the old `select_model` shape did well:

- quick model-backed single selection,
- popup lifetime wiring,
- stable option test-id derivation.

What it does not solve:

- arbitrary row bodies inside the popup,
- a canonical preview/body split equivalent to `BeginCombo()`,
- a clear generic immediate combo story distinct from shadcn/Base UI recipe surfaces.

Latest status:

- canonical `combo(...)` / `combo_with_options(...)` surface now exists in
  `ecosystem/fret-ui-kit::imui`,
- the old `select_model` naming has been removed in favor of combo-aligned authoring,
- a thin `combo_model(...)` convenience remains on top of the canonical combo path for the common
  model-backed single-select case,
- direct interaction gates now cover:
  - Escape close + focus restore on the canonical combo surface
  - selection commit using `selectable(...)` rows inside a canonical combo popup

Recommended owner:

- `ecosystem/fret-ui-kit::imui`

Important boundary:

- this is not the same lane as `docs/workstreams/select-combobox-deep-redesign-v1/`,
- which owns shadcn/Base UI `Select` and `Combobox` part-surface parity.

## P1 gaps

### C. Generic immediate table/columns wrapper

Why it is still a gap:

- Dear ImGui exposes `BeginTable()` as first-class table authoring.
- egui uses both `Grid` and `egui_extras::TableBuilder`.
- Fret has `grid(...)`, but that is a layout container, not a table-oriented immediate helper.

Current consequence:

- dense data panes still have to spell column/header/body structure using lower-level layout
  composition,
- and row-oriented authoring remains less direct than in the reference stacks.

Latest status:

- canonical `table(...)` / `table_with_options(...)` now exists in
  `ecosystem/fret-ui-kit::imui`,
- header/cell vocabulary is intentionally bounded to `TableColumn` + row/cell authoring instead of
  exposing the heavier declarative table surface,
- alignment/clipping gates now cover the immediate helper shape,
- `apps/fret-examples/src/imui_shadcn_adapter_demo.rs` now includes a first-party proof surface.

Recommended owner:

- `ecosystem/fret-ui-kit::imui`

Clarification:

- this should be a bounded table/columns helper,
- not a full data-grid, spreadsheet, or shell tab strip API.

Upstream anchors:

- Dear ImGui: `repo-ref/imgui/imgui.h` (`BeginTable`), `repo-ref/imgui/imgui_tables.cpp`
- egui: `repo-ref/egui/crates/egui/src/grid.rs`,
  `repo-ref/egui/crates/egui_extras/src/table.rs`

### D. Generic list clipper / virtualized immediate list helper

Why it is still a gap:

- Dear ImGui exposes `ImGuiListClipper` for large visible-range submission.
- egui exposes large-list helpers through `ScrollArea::show_rows(...)` and table-body plumbing.
- Fret has editor-specific virtualization in `PropertyGridVirtualized`, but not a generic immediate
  list clipper.

Current consequence:

- large immediate lists still need bespoke visible-range logic,
- and generic `imui` authoring does not yet have a standard answer for large uniform row sets.

Latest status:

- canonical `virtual_list(...)` / `virtual_list_with_options(...)` now exists in
  `ecosystem/fret-ui-kit::imui`,
- the helper reuses the runtime keyed virtual-list substrate instead of pointing callers at the
  editor-only `PropertyGridVirtualized` composite,
- focused gates now cover a deterministic small render window plus scroll-to-target behavior.

Recommended owner:

- `ecosystem/fret-ui-kit::imui`

Evidence:

- generic gap: `ecosystem/fret-ui-kit/src/imui.rs`
- editor-specific proof only: `ecosystem/fret-ui-editor/src/imui.rs`

## P2 gaps

### E. `separator_text` and other tiny hand-feel helpers

Why it is still a gap:

- Dear ImGui exposes `SeparatorText()` as a tiny but heavily reused section-label helper.
- Fret can compose labels + separators manually, but it still lacks the dedicated immediate helper.

Why lower priority:

- this is real authoring friction,
- but it does not unblock selection, combo, table, or large-list ergonomics.

Recommended owner:

- `ecosystem/fret-ui-kit::imui`

Upstream anchors:

- Dear ImGui: `repo-ref/imgui/imgui.h` (`SeparatorText`), `repo-ref/imgui/imgui_widgets.cpp`

Latest status:

- canonical `separator_text(...)` / `separator_text_with_options(...)` now exists in
  `ecosystem/fret-ui-kit::imui`,
- focused gates now cover geometry for the section label plus trailing rule layout,
- the first-party immediate proof surface now uses the helper instead of ad hoc text headings.

## Things that should not be pulled into generic `imui`

These may matter elsewhere, but they should not lead this workstream:

- docking tab bars
- workspace shell tab strips
- broad style-stack or window-flag mirroring
- shell-specific choreography
- shadcn/Base UI part-surface redesign
- editor composites already owned by `fret-ui-editor`

Reason:

- they either belong to different owner crates,
- or they are policy/recipe problems,
- or they are already closed in previous `imui` lanes.

## Practical priority order

1. `selectable`
2. generic `begin_combo` / `combo`
3. immediate table/columns
4. generic list clipper / virtualized rows
5. `separator_text`

## Maintainer rule

Before adding a new `imui` helper, ask:

1. does this close a real high-frequency immediate authoring gap, or only increase surface count?
2. is the owner clearly `fret-ui-kit::imui`, `fret-ui-editor::imui`, or another crate entirely?
3. does the new helper replace overlap, or does it create a second canonical path?

If the owner is unclear or the overlap story is weak, do not land the helper yet.
