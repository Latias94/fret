# Diag per-run artifact layout (canonical)

This document defines the **canonical per-run directory layout** under the diagnostics output dir
(by default `target/fret-diag/`). The goal is a stable, transport-neutral layout that tooling can
rely on for packing, linting, and automation.

This is intentionally scoped to **tooling-side artifacts** (what `fretboard diag ...` produces and
consumes), and is compatible with both:

- filesystem transport (native runner writes bundle exports to disk), and
- DevTools WS transport (tooling materializes bundle payloads to disk).

Related trackers:

- Simplification workstream: `docs/workstreams/diag-simplification-v1.md`
- This workstream (v2 hardening + switches): `docs/workstreams/diag-v2-hardening-and-switches-v1/README.md`

## Terms

- **out dir**: a directory where diag artifacts are written (e.g. `target/fret-diag/`).
- **run id**: a monotonic `u64` generated by the runtime/tooling for each script run.
- **per-run dir**: the stable directory for a single run: `<out_dir>/<run_id>/`.
- **bundle export dir**: a (usually timestamp/label-based) directory containing `bundle.json`
  written by the runtime/tooling. `latest.txt` points to the most recent bundle export dir name.

Important: **per-run dirs and bundle export dirs are not the same thing**.

## Canonical per-run directory

Required:

- `<out_dir>/<run_id>/manifest.json` (small, stable, schema v2)
- `<out_dir>/<run_id>/script.result.json` (tooling/runtime result; includes run id + reason codes)

Optional (best-effort, may be missing depending on the run):

- `<out_dir>/<run_id>/trace.chrome.json`
- `<out_dir>/<run_id>/bundle.json` (alias copy of the last dumped bundle artifact)
- `<out_dir>/<run_id>/bundle.schema2.json` (alias copy when schema2 exists)
- sidecars for fast queries (usually written when `bundle.json` is aliased):
  - `<out_dir>/<run_id>/bundle.meta.json`
  - `<out_dir>/<run_id>/bundle.index.json`
  - `<out_dir>/<run_id>/frames.index.json`
  - `<out_dir>/<run_id>/test_ids.index.json`
  - `<out_dir>/<run_id>/test_ids.json`
- chunked bundle payload (preferred for transport-neutral integrity + packing):
  - `<out_dir>/<run_id>/chunks/bundle_json/chunk-000000` (and more)

Example:

```text
target/fret-diag/
  42/
    manifest.json
    script.result.json
    trace.chrome.json            (optional)
    bundle.json                  (optional alias)
    bundle.schema2.json          (optional alias)
    bundle.meta.json             (optional sidecar)
    bundle.index.json            (optional sidecar)
    frames.index.json            (optional sidecar)
    test_ids.index.json          (optional sidecar)
    chunks/
      bundle_json/
        chunk-000000
        chunk-000001
```

## Bundle export directories + `latest.txt`

Bundle exports are still written as “bundle directories” (filesystem-transport native behavior and
WS materialization behavior):

```text
target/fret-diag/
  2026-02-27_01-23-45_ui-gallery-overlay-torture/
    bundle.json
    bundle.schema2.json          (optional)
    bundle.dumped.json           (optional, WS materialization metadata)
```

`target/fret-diag/latest.txt` contains the directory name of the most recently materialized bundle
export (not a run id).

Tooling bridges these worlds by copying the bundle artifact into the per-run dir when possible.

## `manifest.json` (per-run)

`manifest.json` is the canonical “small index” for a per-run directory.

Current shape (v2) includes:

- `schema_version: 2`
- `generated_unix_ms`
- `run_id`
- `paths.script_result_json` (default: `script.result.json`)
- `files[]`: a bounded list of small files with size + optional hash (e.g. `script.result.json`)
- `bundle_json`: optional chunked descriptor
  - `mode: "chunks.v1"`
  - `total_bytes`, `chunk_bytes`, `blake3`
  - `chunks[]`: `{ index, path, bytes, blake3 }`

Conventions:

- Chunk `path` values are **relative** to the per-run dir and use `/` separators (portable across OS).
- `files[]` entries are also relative and hash-checked when `blake3` is present.

## Validation: `diag artifact lint`

Use this when you want to fail fast on artifact drift/corruption, especially for CI and shareable
bundles.

Examples:

- Validate the latest run id directory (uses the current `script.result.json` to resolve run id):
  - `cargo run -p fretboard -- diag artifact lint`
- Validate a specific per-run dir:
  - `cargo run -p fretboard -- diag artifact lint target/fret-diag/42`
- Validate an out dir root (will resolve the latest run id from `<out_dir>/script.result.json`):
  - `cargo run -p fretboard -- diag artifact lint target/fret-diag`

What it validates (v1):

- manifest exists and parses
- manifest schema version and run id fields
- `manifest.files[]` entries exist and match recorded bytes/hash (when present)
- bundle chunk list exists and matches per-chunk bytes/hash + total hash (when present)
- basic sidecar validation (kind/schema_version/warmup_frames) when sidecars exist
- run id consistency: dir name vs manifest vs `script.result.json`
- timestamp sanity checks (missing/zero/future skew; summary mismatch warnings)

## Repair: `diag doctor`

If `manifest.json` indicates complete bundle chunks but `bundle.json` is missing:

- `cargo run -p fretboard -- diag doctor --fix-bundle-json <bundle_dir_or_out_dir> --warmup-frames <n>`

If sidecars are missing/stale:

- `cargo run -p fretboard -- diag doctor --fix-sidecars <bundle_dir_or_out_dir> --warmup-frames <n>`

Note: `doctor` operates primarily on **bundle directories** (the ones containing `bundle.json` /
`bundle.schema2.json`). Per-run sidecars are best-effort today.
