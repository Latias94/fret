# ImUi Identity Browser HTML v1

Status: closed
Last updated: 2026-04-28

## Why This Lane Exists

`imui-id-stack-browser-v1` closed with a browser-ready source model and an opt-in grouped query
surface: `diag query identity-warnings --browser`.

That is enough for automation, but it still leaves authors reading JSON when they need to inspect
duplicate keyed-list hashes or unkeyed reorder drift. This follow-on owns the next smallest user
experience: a self-contained offline HTML artifact generated from captured bundle data.

## Starting Assumptions

- Area: lane status
  - Assumption: `imui-id-stack-browser-v1` stays closed.
  - Evidence: `docs/workstreams/imui-id-stack-browser-v1/CLOSEOUT_AUDIT_2026-04-28.md`.
  - Confidence: Confident.
  - Consequence if wrong: this lane would duplicate or blur a closed source-model/query lane.

- Area: data source
  - Assumption: the HTML view should consume the same browser model as `--browser`, not walk raw
    bundle JSON independently.
  - Evidence: `crates/fret-diag/src/identity_browser.rs`.
  - Confidence: Confident.
  - Consequence if wrong: grouping/filter semantics could drift from the CLI query.

- Area: user journey
  - Assumption: a local HTML sidecar is the first useful interactive artifact.
  - Evidence: closeout decision in `imui-id-stack-browser-v1` explicitly deferred dashboard/live
    panels and kept `--browser --json --out` as the stable input shape.
  - Confidence: Likely.
  - Consequence if wrong: dashboard integration should be split into a narrower follow-on after the
    HTML renderer proves the review shape.

- Area: policy boundary
  - Assumption: HTML rendering is diagnostics presentation only; it must not infer `test_id`s,
    rewrite identity contracts, or introduce public runtime APIs.
  - Evidence: ADR 0319 and the closed browser workstream boundaries.
  - Confidence: Confident.
  - Consequence if wrong: start a contract lane before adding new runtime fields or authoring APIs.

## Target Surface

The first HTML artifact should be:

- self-contained and safe to open from disk,
- generated by `fret-diag` from a schema2 bundle or existing query data,
- dense and diagnostic-focused rather than marketing-like,
- searchable/filterable by kind, source file, element path, list id, and key hash,
- grouped by the browser model's stable group key,
- linked to the source bundle path and warning anchors already present in JSON.

## Non-Goals

- No live connected devtools panel.
- No dashboard framework integration in the first slice.
- No public runtime identity API.
- No label-to-`test_id` inference.
- No localization or table column identity changes.

## Exit Criteria

- A focused HTML renderer exists with escaping/fixture tests.
- `diag query identity-warnings` can write a self-contained HTML sidecar without breaking existing
  JSON output.
- Workstream evidence names the renderer, command surface, tests, and gates.
