# Fret v0.1.0 Release Checklist

Date prepared: 2026-02-27.

This checklist is the operational runbook for the first `release-plz`-driven publish wave.

## 1) Scope and config freeze

- Confirm `release-plz.toml` is the single source of truth for the publish whitelist.
- Keep workspace defaults conservative (`release = false`, `publish = false`).
- Keep all v0.1 crates in the same `version_group` (`fret-0-1`).
- Keep the user-facing release story centered on:
  - `fret`
  - `fret-framework`
  - `fret-bootstrap`
  - `fret-ui-kit`
  - `fret-ui-shadcn`
  - `fret-selector`
  - `fret-query`
- Keep tooling-only crates and deferred ecosystems out of the first public teaching surface.

## 2) Local closure and order checks

- Run release scope closure check:
  - `python tools/release_closure_check.py --write-order docs/release/v0.1.0-publish-order.txt --write-waves docs/release/v0.1.0-publish-waves.txt --print-publish-commands --print-publish-waves`
- Before any local `release-plz update` preflight, install the checkout override block:
  - `python3 tools/prepare_release_plz_checkout_overrides.py`
- For later publish waves, generate a bounded `cargo publish --dry-run` report instead of triaging crates one by one:
  - `python3 tools/release_wave_dry_run.py --wave <N> --allow-registry-gap`
- Expected:
  - `release scope: <N> crates` (keep `docs/release/v0.1.0-publish-order.txt` in sync)
  - `publish waves: <N>` (keep `docs/release/v0.1.0-publish-waves.txt` in sync)
  - `internal dependency issues: 0`
  - deterministic publish order output
  - metadata warnings list (non-blocking unless strict mode is enabled)
- Current preflight status on 2026-04-01:
  - `release scope: 49 crates`
  - `publish waves: 12`
  - `wave 1: fret-assets, fret-core, fret-i18n, fret-icons, fret-render-core`
  - `internal dependency issues: 0`
  - `metadata warnings: 0`
  - `docs/release/v0.1.0-publish-order.txt` regenerated from the current config
  - `docs/release/v0.1.0-publish-waves.txt` regenerated from the current config
  - `docs/release/v0.1.0-wave-1-dry-run.txt` confirms all five wave 1 crates pass `cargo publish --dry-run`
  - `docs/release/v0.1.0-wave-2-dry-run.txt` through `docs/release/v0.1.0-wave-12-dry-run.txt` record that every later wave is packaging-clean and only blocked by earlier waves not yet being visible on crates.io
  - `docs/release/v0.1.0-release-plz-preflight.txt` records that package-scoped and full-workspace `release-plz update` planning now pass with the checkout override helper

Optional strict metadata gate:

- `python tools/release_closure_check.py --strict-metadata`

## 3) CI workflow readiness

- Validate `.github/workflows/release-plz.yml` includes:
  - `fetch-depth: 0`
  - `python3 tools/prepare_release_plz_checkout_overrides.py` before `release-plz`
  - separate `release-pr` and `release` jobs
  - `config: release-plz.toml`
  - canonical repo guard (`github.repository == 'Latias94/fret'`)
- Repository settings and secrets:
  - allow GitHub Actions to create PRs
  - set `CARGO_REGISTRY_TOKEN`
  - keep default `GITHUB_TOKEN` permissions aligned with workflow job permissions

## 4) Release PR stage

- Trigger `release-plz` `release-pr` (via push to `main` or `workflow_dispatch`).
- Review generated release PR:
  - version bumps are expected
  - changelog/release notes are acceptable
  - no unexpected crate in scope
- Merge release PR to `main`.

## 5) Publish stage

- After release PR merge, trigger/observe `release-plz` `release` job.
- Monitor publish sequence and crates.io visibility.
- Use `docs/release/v0.1.0-publish-waves.txt` to reason about which local dry-runs are expected to fail until earlier waves propagate to crates.io.
- Use `python3 tools/release_wave_registry_status.py --wave <N>` to confirm when a completed wave is actually visible on crates.io before retrying a later-wave failure.
- Regenerate `docs/release/v0.1.0-wave-<N>-dry-run.txt` with `python3 tools/release_wave_dry_run.py --wave <N> --allow-registry-gap` when you need fresh evidence for a later wave.
- If a publish fails because of dependency availability lag, retry after crates index propagation.

## 6) Post-release verification

- Verify tags/releases were created by automation.
- Spot-check crates on crates.io (especially top-level surfaces):
  - `fret`
  - `fret-framework`
  - `fret-bootstrap`
  - `fret-ui-kit`
  - `fret-ui-shadcn`
  - `fret-selector`
  - `fret-query`
- Run a quick consumer sanity check in a clean sample project.

## 7) Known risk notes (v0.1)

- `cargo publish --dry-run -p <crate>` for high-level crates may fail locally before dependencies are actually published to crates.io.
- This is expected for first-wave multi-crate publish and should be evaluated against publish waves, not as an isolated crate failure.
- `wave 1` crates are the only ones expected to dry-run cleanly before any Fret crate is present on crates.io; later waves require earlier waves to be visible first.
- During the actual publish run, use `python3 tools/release_wave_registry_status.py --wave <N> --require-visible` to distinguish crates.io propagation lag from a real publish blocker.
- Running `release-plz update` without the checkout override helper can fail during historical diff recovery because older shadcn source blobs and ADR audit docs carry CRLF content under `*.rs` / `*.md` `eol=lf` rules. Install the managed `.git/info/attributes` block via `python3 tools/prepare_release_plz_checkout_overrides.py` before local planning and keep the workflow step in place. With that helper installed, package-scoped and full-workspace `release-plz update` planning are currently validated. See `docs/release/v0.1.0-release-plz-preflight.txt`.
- `package.description` has been populated for the current v0.1 release scope; keep descriptions aligned with crate role as APIs evolve.
- MSRV is intentionally pinned to Rust `1.92` (workspace `rust-version`) to match current `wgpu` minimum requirements.
