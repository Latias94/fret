# Fret v0.1.0 Release Checklist

Date prepared: 2026-02-08.

This checklist is the operational runbook for the first `release-plz`-driven publish wave.

## 1) Scope and config freeze

- Confirm `release-plz.toml` is the single source of truth for the publish whitelist.
- Keep workspace defaults conservative (`release = false`, `publish = false`).
- Keep all v0.1 crates in the same `version_group` (`fret-0-1`).
- Confirm `fret-node` and `fret-router` are included for v0.1 by product decision.

## 2) Local closure and order checks

- Run release scope closure check:
  - `python tools/release_closure_check.py --write-order docs/release/v0.1.0-publish-order.txt --print-publish-commands`
- Expected:
  - `internal dependency issues: 0`
  - deterministic publish order output
  - metadata warnings list (non-blocking unless strict mode is enabled)

Optional strict metadata gate:

- `python tools/release_closure_check.py --strict-metadata`

## 3) CI workflow readiness

- Validate `.github/workflows/release-plz.yml` includes:
  - `fetch-depth: 0`
  - separate `release-pr` and `release` jobs
  - `config: release-plz.toml`
  - canonical repo guard (`github.repository == 'Latias94/fret'`)
- Repository settings and secrets:
  - allow GitHub Actions to create PRs
  - set `CARGO_REGISTRY_TOKEN`
  - keep default `GITHUB_TOKEN` permissions aligned with workflow job permissions

## 4) Release PR stage

- Trigger `release-plz` `release-pr` (via push to `main` or `workflow_dispatch`).
- Review generated release PR:
  - version bumps are expected
  - changelog/release notes are acceptable
  - no unexpected crate in scope
- Merge release PR to `main`.

## 5) Publish stage

- After release PR merge, trigger/observe `release-plz` `release` job.
- Monitor publish sequence and crates.io visibility.
- If a publish fails because of dependency availability lag, retry after crates index propagation.

## 6) Post-release verification

- Verify tags/releases were created by automation.
- Spot-check crates on crates.io (especially top-level surfaces):
  - `fret`
  - `fret-ui-kit`
  - `fret-ui-shadcn`
  - `fret-node`
  - `fret-router`
- Run a quick consumer sanity check in a clean sample project.

## 7) Known risk notes (v0.1)

- `cargo publish --dry-run -p <crate>` for high-level crates may fail locally before dependencies are actually published to crates.io.
- This is expected for first-wave multi-crate publish and should be evaluated against publish order, not as an isolated crate failure.
- `package.description` has been populated for the current v0.1 release scope; keep descriptions aligned with crate role as APIs evolve.
- MSRV is intentionally pinned to Rust `1.92` (workspace `rust-version`) to match current `wgpu` minimum requirements.
